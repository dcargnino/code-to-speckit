# Code-to-SpecKit Config

## Mode

brownfield

## Obsidian Policy

- Add frontmatter and tags to generated Markdown in `docs/reverse-engineering/` and `specs/`.
- Keep `.code-to-speckit/` guidance files lightweight and editable.

## Evidence Policy

- Do not promote Unknown claims into requirements.
- Require at least one direct evidence reference per functional requirement.
- Prefer file path plus line, symbol, route, test, or schema name when available.

## Large Repo Policy

- Require navigation, index, inventory, scope, trace, and evidence before generating a spec.
